LinkedBoost - Complete Code Audit Report
EXECUTIVE SUMMARY
I analyzed every file across all core modules. The UI shell and API surface are well-crafted, but there are system-breaking integration gaps that prevent the automation from functioning end-to-end. Below is everything organized by priority.

P0 - CRITICAL (System Non-Functional Without Fixes)
1. WebSocket Protocol Mismatch - Extension Cannot Connect to Server
Files: extension/background/service-worker.js:29, lib/websocket/server.ts:129

The extension uses raw new WebSocket(wsUrl) but the server uses Socket.IO (which requires its own client library with HTTP handshake + packet framing). These are incompatible protocols. The extension literally cannot connect to the server. The entire real-time pipeline (job scraping, posting, scraping results) is non-functional.

Fix: Either replace Socket.IO server with a raw WebSocket server (e.g., ws library), or add socket.io-client to the extension's service worker.

2. No Orchestration Engine - Nothing Chains Actions Together
The individual building blocks exist (content script commands, API endpoints, AI services) but nothing orchestrates them into automated workflows:

No code translates JobSearch configs into LinkedIn URLs, navigates there, scrapes, scores, tailors resumes, fills forms, and submits
No code executes scheduled posts at their scheduled time
No code translates ScraperConfig records into extension commands
Each step exists in isolation
Fix: Build an orchestration service (either server-side with task queues or extension-side with state machines) that chains the individual commands into workflows.

3. OAuth Users Get Wrong User ID (BUG-AUTH-3)
File: auth.ts:60-73, auth.config.ts:63

When Google/GitHub users sign in, token.id is set to the OAuth provider's user ID (e.g., Google's "117...") instead of the MongoDB _id. All subsequent API calls using session.user.id fail to match any MongoDB document. OAuth users can sign in but cannot use any feature.

Fix: In the signIn callback, after creating/finding the user, look up the MongoDB user and set the token ID properly:

4. Content Script Navigation Bug Destroys Execution Context
File: extension/content/content-script.js:595-598

createLinkedInPost sets window.location.href when not on /feed, which causes a full page reload that destroys the content script's execution context. The post is never created. Same issue would affect any command that navigates.

Fix: Navigation should be handled by the service worker via chrome.tabs.update, then re-send the command after navigation completes.

5. PDF Generator Completely Missing
The plan calls for lib/services/pdf-generator.ts to generate tailored resume PDFs. This file does not exist. Without it, UPLOAD_RESUME has no server-generated PDF to upload, breaking the entire Easy Apply flow.

6. Anti-Detection Engine Is Dead Code
All files in lib/anti-detection/ contain sophisticated server-side code (Gaussian delays, session tracking, safety scores) that is never imported or used by the content script. The content script uses hardcoded flat delays (500ms, 1000ms, 2000ms, 3000ms). The entire anti-detection sophistication is unreachable.

P1 - HIGH (Incorrect Results / Data Corruption)
7. tailoredResume Field Name Mismatch
Files: lib/services/job-analysis.ts:130-134 vs components/jobs/jobs-client.tsx:69-73

Backend stores { summary, skills, highlights } but frontend reads { tailoredSummary, tailoredSkills, matchExplanation }. All tailored resume UI shows nothing.

8. Stats API Field Name Mismatch
job-analysis.ts returns averageMatchScore but frontend reads avgMatchScore → Always shows NaN%
Applications API returns no total field but frontend reads data.total → Always shows 0
9. MongoDB Aggregation userId Type Mismatch
Files: app/api/scraper/route.ts:103-117, lib/services/job-analysis.ts:221

Aggregation pipelines receive raw string session.user.id but userId is stored as ObjectId. Mongoose auto-casts in find() but NOT in aggregation $match. Stats always return empty/zero.

Fix: { $match: { userId: new mongoose.Types.ObjectId(session.user.id) } }

10. Gemini generateJSON Crashes on Safety-Blocked Content
File: lib/ai/providers/gemini.ts:55

When Gemini blocks content, generateText returns "", which flows into JSON.parse("") → crash. Also doesn't use Gemini's native responseMimeType: "application/json".

11. Gemini API Key in URL Query String
File: lib/ai/providers/gemini.ts:41

API key passed as URL parameter (visible in logs, proxies, error tracking). OpenAI/Groq correctly use Authorization header.

12. Rate Limiter Off-By-One
File: lib/utils/rate-limit.ts:24-46

Uses > instead of >=, allowing 6 requests when limit is 5. Also has a race condition between find and create operations.

13. Hero Posting Days Buttons Non-Functional
File: components/hero/hero-client.tsx:335-349

Day-selection buttons have no onClick handler. The posting schedule days can never be changed by the user.

14. Content Script Profile/Post/Company Scraping Missing
The content script only has SCRAPE_JOB_LISTINGS and SCRAPE_JOB_DETAIL but ScraperConfig supports types posts, profiles, companies. There are no corresponding commands.

15. saveApiKey Non-Atomic Race Condition
File: lib/ai/key-manager.ts:35-47

Two separate MongoDB operations ($pull + $push) can race, leaving zero or duplicate keys.

16. OTP Brute-Force Vulnerability
6-digit OTP (900K combinations) with only IP-based rate limiting. No per-user/per-email rate limiting. Distributed attack feasible.

17. Password Reset Token Stored in Plaintext
File: app/api/auth/forgot-password/route.ts

Reset tokens stored unencrypted in MongoDB. Database breach = instant password reset for any user.

18. WebSocket Auth Uses Raw userId as "Token"
File: lib/websocket/server.ts:155

Anyone who knows a MongoDB ObjectId can impersonate that user's extension connection.
